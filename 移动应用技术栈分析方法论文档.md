## 一、分析方法概述

### 核心原则
1. **证据为王**: 只报告有实际证据支持的技术
2. **多维度验证**: DEX + SO + assets 三者交叉验证
3. **诚实标注**: 对不确定内容明确标注置信度
4. **避免推断**: 不基于应用类型或同类应用推测技术栈

### 分析流程
```
目录结构 → DEX分析 → Native库分析 → Assets检查 → META-INF分析 → 交叉验证 → 报告生成
```

## 二、分析手段详解

### 1. DEX文件字符串分析

**目的**: 识别Java/Kotlin类名、方法签名、包名

**工具**: `strings` 命令

**命令示例**:
```bash
# 提取所有可读字符串
strings classes.dex | grep -i "framework_name"

# 查找特定框架的包名
strings classes.dex | grep "Lcom/facebook/react"

# 查找特定类
strings classes.dex | grep "Lio/flutter/embedding"

# 输出到文件
strings classes*.dex > dex_strings.txt
```

**关键框架特征字符串**:
| 框架 | 特征字符串 |
|------|-----------|
| Flutter | `Lio/flutter/`, `FlutterEngine`, `MethodChannel` |
| React Native | `Lcom/facebook/react/`, `ReactPackage`, `NativeModule` |
| Weex | `Lcom/taobao/weex`, `Lcom/alibaba/aliweex` |
| WindVane | `Landroid/taobao/windvane`, `WindVaneSDK` |
| Uni-App | `Lio/dcloud`, `uni-jsframework` |
| KMP | `Lkotlin/native`, `kotlinx.cinterop` |
| Jetpack Compose | `Landroidx/compose/` |

**证据价值**: ⭐⭐⭐⭐⭐ (最高)

---

### 2. Native库（.so）文件分析

**目的**: 识别C/C++框架、引擎、第三方库

**工具**: `file` + `strings` + `ls -lh`

**命令示例**:
```bash
# 查看所有SO文件
ls -lh lib/arm64-v8a/*.so

# 检查SO文件类型
file lib/arm64-v8a/libflutter.so

# 提取SO中的符号表
strings lib/arm64-v8a/libflutter.so | grep -i "version"

# 查找特定框架特征
strings lib/arm64-v8a/libweexcore.so | head -20
```

**关键SO文件识别**:
| SO文件 | 框架/用途 |
|--------|----------|
| libflutter.so (9.6MB) | Flutter引擎 |
| libapp.so | Dart快照 |
| libreactnative.so | React Native核心 |
| libhermes.so | Hermes JS引擎 |
| libweexcore.so | Weex核心 |
| libquickjs.so | QuickJS引擎 |
| libmmkv.so | MMKV存储 |

**证据价值**: ⭐⭐⭐⭐⭐ (最高)

---

### 3. Assets目录检查

**目的**: 发现JavaScript框架、资源文件、配置文件

**工具**: `find` + `ls` + `file`

**命令示例**:
```bash
# 列出assets目录结构
find assets -type f | head -50

# 查看JS框架文件
ls -lh assets/*.js

# 检查特定文件
file assets/uni-jsframework.js

# 查找Flutter资源
find assets -name "flutter_assets" -type d
```

**关键文件识别**:
| 文件/目录 | 框架/用途 |
|-----------|----------|
| flutter_assets/ | Flutter资源 |
| uni-jsframework.js | Uni-App Vue2 |
| uni-jsframework-vue3.js | Uni-App Vue3 |
| cflutter.info | Flutter配置 |
| bundle.js | React Native bundle |
| dcloud_uniplugins.json | Uni-App插件 |

**证据价值**: ⭐⭐⭐⭐⭐ (最高)

---

### 4. META-INF目录分析

**目的**: 获取版本信息、签名信息、服务声明

**工具**: `cat` + `grep`

**命令示例**:
```bash
# 查看MANIFEST.MF
cat META-INF/MANIFEST.MF

# 查看所有签名文件
ls META-INF/*.RSA

# 查找特定框架元数据
find META-INF -name "*.kotlin_module" -o -name "kotlin*"

# 检查服务声明
ls META-INF/services/
```

**关键信息识别**:
| 文件/目录 | 信息类型 |
|-----------|----------|
| MANIFEST.MF | JAR清单 |
| *.RSA / *.SF | 签名信息 |
| kotlin* | Kotlin版本 |
| services/ | 服务声明 |
| litho/ | Litho框架元数据 |
| mmPaaS/ | 美团mPaaS元数据 |

**证据价值**: ⭐⭐⭐⭐ (高)

---

### 5. AndroidManifest.xml解析

**注意**: 需要使用 `aapt` 或 `apktool` 解码

**工具**: `aapt`

**命令示例**:
```bash
# 解码AndroidManifest.xml
aapt dump xmltree app.apk AndroidManifest.xml

# 查看包名和版本
aapt dump badging app.apk | grep package

# 查看所有组件
aapt dump xmltree app.apk AndroidManifest.xml | grep "activity\|service\|receiver"
```

**证据价值**: ⭐⭐⭐ (中)

---

### 6. 目录结构分析

**目的**: 了解应用组织方式，识别框架特征

**工具**: `tree` 或 `find`

**命令示例**:
```bash
# 显示完整目录树
tree -L 3 -d

# 查找特定目录
find . -type d -name "META-INF" -o -name "assets" -o -name "lib"

# 统计文件数量
find . -type f | wc -l
```

**关键目录识别**:
| 目录 | 用途 |
|------|------|
| lib/ | Native库 |
| assets/ | 资产文件 |
| res/ | 资源文件 |
| META-INF/ | 元数据 |
| kotlin/ | Kotlin代码 |
| root/ | 某些框架使用 |

**证据价值**: ⭐⭐⭐ (中)

## 三、框架识别特征库

### Flutter识别
```
✅ DEX: Lio/flutter/embedding/engine/
✅ SO: libflutter.so (通常9.6MB)
✅ SO: libapp.so
✅ assets: flutter_assets/
✅ assets: .flutter-plugins-dependencies
```

### React Native识别
```
✅ DEX: Lcom/facebook/react/
✅ SO: libreactnative.so
✅ SO: libhermes.so (如果使用Hermes)
✅ assets: index.android.bundle
```

### Weex识别
```
✅ DEX: Lcom/taobao/weex 或 Lcom/alibaba/aliweex
✅ SO: libweexcore.so
✅ SO: libweexjsb.so
```

### Uni-App识别
```
✅ DEX: Lio/dcloud/
✅ assets: uni-jsframework.js
✅ SO: libweexcore.so (底层)
✅ assets: dcloud_uniplugins.json
```

### WindVane识别
```
✅ DEX: Landroid/taobao/windvane
✅ DEX: __windvane__.nativeCall
✅ 证据: WVCacheManager, WVPerformanceManager
```

### Kotlin Multiplatform (KMP)识别
```
✅ DEX: Lkotlin/native/
✅ META-INF: .klib 文件
⚠️ 注意: 仅有Kotlin代码不等于KMP
```

### Jetpack Compose识别
```
✅ DEX: Landroidx/compose/
✅ res: composeResources/ 目录
⚠️ 注意: composeResources目录是强证据，但需要其他证据支持
```

## 四、证据等级系统

### 一级证据（确凿）
- DEX中的完整类名
- SO文件的特征库
- assets中的框架文件
- **置信度**: 95%+

### 二级证据（强）
- META-INF中的元数据
- 目录结构的特征
- 相关的配置文件
- **置信度**: 80%+

### 三级证据（中等）
- 单一的文件名
- 部分的字符串匹配
- 间接关联
- **置信度**: 60%+

### 四级证据（弱）
- 基于应用类型的推断
- 同类应用对比
- 缺乏直接证据
- **置信度**: <50%（不应报告）

## 五、常见错误及避免方法

### 错误1: 仅凭包名推测
```
❌ 错误: 包名含"pay"就断定使用支付宝SDK
✅ 正确: 查找"com.alipay"的具体类引用
```

### 错误2: 混淆框架功能和应用使用
```
❌ 错误: Uni-App有支付功能 → 应用使用了支付
✅ 正确: 查找应用实际的支付SDK集成证据
```

### 错误3: 过度推断
```
❌ 错误: 携程是OTA → 应该使用KMP
✅ 正确: 查找KMP的实际证据（无证据则不报告）
```

### 错误4: 单一证据依赖
```
❌ 错误: 仅有composeResources目录就断定使用Compose
✅ 正确: 同时查找DEX中的androidx.compose类引用
```

### 错误5: 忽略反面证据
```
❌ 错误: 声称使用KMP，但无视无kotlin.native的证据
✅ 正当: 面对矛盾证据时，以直接证据为准
```

## 七、质量控制检查清单

在提交分析报告前，检查以下问题：

### 证据完整性
- [ ] 每个关键技术栈声明都有证据支持
- [ ] 证据来源明确标注（DEX/SO/assets/META-INF）
- [ ] 重要结论有多个独立证据支持

### 准确性验证
- [ ] 没有基于应用类型的推断
- [ ] 没有虚构的模块名称或类名
- [ ] 区分了框架功能和应用实际使用

### 诚实性标注
- [ ] 不确定的内容标注了置信度
- [ ] 可能的定制版本有说明
- [ ] 分析局限性有提及

### 格式规范
- [ ] 包含数据来源声明
- [ ] 证据块清晰可读
- [ ] 修正说明完整（如适用）

## 九、案例研究

### 案例1: Flutter识别（完整证据链）

**闲鱼应用**:
```
1. DEX证据：
   - io/flutter/embedding/engine/plugins/FlutterPlugin
   - io/flutter/plugin/common/MethodChannel

2. SO证据：
   - libflutter.so (9.6MB)
   - libapp.so (Dart快照)

3. assets证据：
   - flutter_assets/目录
   - Flutter字体文件
   - Flutter配置文件

结论：✅ 确认使用Flutter（高置信度）
```

### 案例2: 错误识别的案例

**携程应用 - KMP**:
```
原报告声称：
- 使用Kotlin Multiplatform
- 模块：KMPFoundation, KMPNetwork等

实际证据：
- ❌ DEX中无kotlin.native包名
- ❌ SO中无KMP特征库
- ❌ 无.klib文件
- ✅ 仅有标准Kotlin代码

结论：❌ 未使用KMP（高置信度否定）
```

### 案例3: 部分使用的识别

**千问应用 - Flutter**:
```
发现证据：
- ⚠️ libunet.so中有unet_flutter_plugin字符串
- ⚠️ libunet.so中有UCFlutter字符串
- ❌ 但无libflutter.so主引擎
- ❌ 但无libapp.so应用快照

结论：⚠️ 可能使用Flutter作为插件/模块（中置信度）
      但不是主要UI框架
```

## 十、持续改进

### 已知局限性
1. 加固/混淆的应用分析困难
2. 动态加载的代码可能不在初始DEX中
3. 某些框架可能完全重命名包名

### 改进方向
1. 建立更完整的框架特征库
2. 开发自动化验证脚本
3. 收集更多真实应用案例

### 最佳实践
1. 对可疑声明进行二次验证
2. 保存分析过程中的原始数据
3. 建立同行评审机制
